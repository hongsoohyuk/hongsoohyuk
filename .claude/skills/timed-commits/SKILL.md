변경사항을 논리적으로 나눠서 특정 시간대에 커밋을 생성합니다.

## 입력 파싱

$ARGUMENTS에서 날짜와 시간 범위를 추출하세요:
- "3월 16일 19시~23시" → 2025-03-16, 19:00~23:00
- "03-16 19~23" → 2025-03-16, 19:00~23:00
- "2025-03-16 19:00-23:00" → 그대로 사용
- 날짜가 없으면 오늘 날짜 사용
- 시간이 없으면 19:00~23:00 기본값

## 워크플로우

1. `git status`와 `git diff --stat`으로 변경사항 파악
2. 변경사항을 **기능/목적별로 논리적 그룹**으로 나누기 (보통 2~5개)
3. 각 그룹에 적절한 커밋 메시지 작성 (프로젝트 기존 커밋 스타일 참고)
4. 사용자에게 커밋 계획을 보여주고 확인 받기
5. 확인 후 커밋 실행

## 커밋 시간 생성 규칙

- 시작시~종료시 사이에 **랜덤한 분/초**로 생성
- 시간 순서는 커밋 순서와 일치해야 함
- 분은 0~59 중 랜덤, 초는 0~59 중 랜덤
- 예) 19:17:43, 20:04:11, 21:38:52, 22:51:06
- **절대 00분, 30분 같은 정각으로 만들지 않기**
- 커밋 간 최소 15분 간격 유지

## 커밋 실행

```bash
GIT_AUTHOR_DATE="YYYY-MM-DDTHH:MM:SS+09:00" GIT_COMMITTER_DATE="YYYY-MM-DDTHH:MM:SS+09:00" git commit -m "메시지"
```

- 타임존은 KST (+09:00) 고정
- 커밋 메시지는 프로젝트 컨벤션 따르기 (feat/fix/refactor/style/chore 등)
- 커밋 후 `git log --oneline`으로 결과 확인

## 커밋 그룹 나누는 기준

- **기능별**: 새 기능, 버그 수정, 리팩터링, 스타일 변경
- **모듈별**: 서로 다른 feature 폴더의 변경은 분리
- **관련성**: 같은 목적의 변경은 하나로 묶기
- 설정 파일(.env, config 등)은 관련 기능 커밋에 포함

## 주의사항

- staged 파일이 있으면 해당 파일만 대상으로 함
- push는 사용자가 명시적으로 요청할 때만 수행
- 커밋 계획을 먼저 보여주고 "이대로 진행할까요?" 확인 받기
